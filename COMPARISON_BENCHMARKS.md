## What Aethelix Actually Does Well

Aethelix is a **zero-shot, causal inference engine**. Its advantages over
LSTM-based detectors are:

| Advantage | Detail |
| :--- | :--- |
| **Zero training data** | Causal graph is physics-derived; no historical data required |
| **Explainability** | Returns ranked causal paths, not just "anomaly detected" |
| **Contextual suppression** | Eclipse-awareness eliminates spurious solar-channel alerts |
| **Sub-threshold sensitivity** | Detects distribution shifts below fixed-limit alarm thresholds |

Aethelix does **not** claim to match a trained LSTM on raw F1 — that is
LSTM's home turf. The comparison below is honest about this.

## 📊 NASA SMAP / MSL Benchmark

### Evaluation Protocol (sequence-level, matching Hundman et al. 2018)

- **True Positive (TP):** At least one alarm fires inside a labelled anomaly window.
- **False Positive (FP):** An alarm fires with no overlap to any anomaly window.
  Consecutive alarms in the same non-anomaly region count as **ONE FP event**.
- **False Negative (FN):** A labelled anomaly window receives zero alarm overlap.

> [!NOTE]
> Raw sample-level accuracy is not used here because a single unlucky sample
> in a ~8 500-sample channel would produce thousands of FP "samples" while
> counting as only one FP event. Sequence-level evaluation is the standard.

| :--- | :--- | :--- | :--- |

| **Recall** | *see run output* | ~85% | ~53% |
| **F1 Score** | *see run output* | ~85% | ~37% |
| **FP events / channel** | *see run output* | N/A (trained) | High |
| **Training required** | **None** | Days–weeks | None |
| **Explainability** | **Causal paths** | None (black box) | Alert only |

> [!WARNING]
> The SMAP/MSL dataset contains predominantly **contextual anomalies** —
> subtle pattern-regime shifts. Univariate z-score detectors produce
> ~0.3% false-trigger rate per sample at z=3.0, yielding ~25 spurious
> events per 8 500-sample channel before non-stationarity is accounted for.
> Aethelix uses a dual-window KS-test (`p=0.005`, `persist=4`) which
> directly targets distribution shifts, producing far fewer FP events at
> the cost of some recall on point anomalies.

**Honest context:**
The LSTM Telemanom baseline (85% F1) is trained on the full SMAP/MSL
training split. Aethelix is genuinely zero-shot. The expected trade-off is
lower raw F1 for Aethelix in exchange for no training overhead and full
causal explainability on every alarm.

## 🔬 Sub-threshold Fault Detection (5–12% Severity)

Most satellite failures begin as subtle drifts below standard 15% alarm
thresholds. This benchmark measures whether Aethelix can detect them before
they reach the threshold level.

### Methodology (`scripts/subthreshold_benchmark.py`)

- **100 scenarios**, `seed=42`, solar degradation drawn from `Uniform(0.05, 0.12)`
- Fault injected at T+6 h; detection window = 2 h post-onset
- **Confidence threshold: ≥ 40%** (meaningful, not trivially low)
- Correct detection requires `top-1 hypothesis == "solar_degradation"` with ≥ 40% confidence
- False-positive rate measured on **30 separate clean-data scenarios** (no fault injected)

| :--- | :--- | :--- | :--- |

| **False positive rate (clean data)** | *see run output* | ~5% | 0% |
| **Mean lead time from fault onset** | *see run output* | ~15 s | N/A (never fires) |
| **Training required** | **None** | High | None |

**Honest context:**
The traditional 15% threshold alarm misses **all** faults in this severity
range by design. LSTM detection in this regime is noise-floor limited
(~35%). Aethelix uses causal path correlation to identify consistent
multi-channel anomaly patterns, enabling detection below the threshold.

## ⏱️ Detection Lead-Time Benchmark

### Definition (`scripts/streaming_benchmark.py`)

```
Lead time = t_threshold_alarm – t_aethelix_detection
```

Where:
- `t_aethelix_detection` = first sample where Aethelix produces the correct
  top-1 hypothesis with confidence ≥ 40%
- `t_threshold_alarm` = first sample where channel deviation exceeds 15% of
  nominal mean (OOL trigger)

A positive lead time means Aethelix detects **earlier** than the threshold
alarm. For sub-15% faults, the threshold never fires, so lead time is
effectively `+∞` (Aethelix-only detection).

### Methodology

- **50 scenarios**, `seed=42`, solar degradation 15–40% injected at T=6 h
- Sample rate: 0.1 Hz (1 sample / 10 seconds)
- Both detectors observe **the same sample stream simultaneously**

> [!NOTE]
> The lead-time metric measures **prediction advantage** — how many seconds
> earlier Aethelix correctly identifies the fault compared to the threshold
> alarm on the **same data stream**. It does **not** measure pipeline

**Honest context:**
For faults comfortably above 15%, both Aethelix and the threshold alarm
fire within seconds of each other. The lead-time advantage grows for faults
near or below the 15% boundary, where the threshold alarm delays or fails
entirely. LSTM Telemanom requires training before deployment; its lead-time
advantage derives from learned prediction rather than causal graphs.

## 🛡️ Reliability & Robustness

### Eclipse Awareness (False Positive Suppression)

Traditional anomaly detectors often trigger "low power" alarms during
orbital eclipse — a predictable, non-fault event.

- **Approach:** Aethelix suppresses solar-coupled channels (`solar_input`,
  `solar_panel_temp`, `battery_charge`, `bus_voltage`) during the eclipse
  window (orbital phase 0.15–0.85).
- **Result:** Zero spurious eclipse alarms on clean orbital data.
- **Threshold/LSTM baselines:** Typically 10–20 false alarms per orbital transition.

### Sensor Fault Isolation

Aethelix distinguishes a sensor failure from a physical system failure:

- **Test:** Single dead sensor (`battery_voltage = 0.0`) during nominal operations.
- **Result:** Aethelix classifies this as `SENSOR_FAULT` with high confidence,
  not a "Battery Critical" system-wide alert.
